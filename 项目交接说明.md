# Flight Flow 项目交接说明

这个项目是一个给 iPad 使用的飞行阶段提醒 PWA。它面向 A320 运行 / 模拟机训练场景，用于按飞行阶段显示易忘项目、风险提示、环境条件触发提示和模拟机科目提示。

https://tianming-798.github.io/flight-flow/

GitHub 仓库：

https://github.com/tianming-798/flight-flow

```powershell
C:\Users\13506\Documents\运行进程
```

## 当前已实现的重点功能

### 阶段进程

- 当前阶段模式
- 总览模式
- 上一阶段 / 下一阶段
- 已完成阶段显示
- 存在未勾选易忘项目时，进入下一阶段会弹出遗漏确认
- 左侧阶段列表和总览卡片只用于查看 / 预览，不会提前标记阶段完成
- 查看非当前执行阶段时，底部显示“回到当前阶段”
- 查看非当前执行阶段时，易忘项目不能被误勾选
- 只有底部“完成并进入下一阶段”才会推进当前阶段和完成状态

### 环境条件

环境条件和飞机信息已经拆成两个独立入口。

“环境条件”只包含天气和跑道相关字段：

- 机场
- 气温
- 风向
- 风速
- 阵风
- 能见度
- 是否降水
- 是否需要除 / 防冰
- 跑道
- 跑道方向
- 跑道代码
- 刹车效应
- 备注

所有字段都是选填。空字段不参与规则判断，不报错，不阻止阶段推进。

运行方式按气温自动判断：

- 未填写气温：未判定
- 气温 > 30℃：高温运行
- 气温 ≤ 0℃：冬季运行
- 其他：常规运行

如果“高温运行”等环境规则已经命中，顶部不会再重复显示同名运行方式标签。

### 飞机信息

“飞机信息”只包含飞机和重量相关字段：

- 机型
- 油量，单位吨
- 0 燃油重量，单位吨

飞机信息页会显示预计起飞重量：

```text
预计起飞重量 = 油量 + 0 燃油重量
```

当前这个数值只是显示预览，后续可用于大重量起飞风险规则。

### 已内置的环境规则

规则在 `src/data.ts` 的 `defaultRules` 中维护。

- 高温运行
  - 条件：气温 > 30℃
  - 驾驶舱准备：易忘项目“黄+蓝超压泵放襟翼 1”

- 冬季运行（模板）
  - 默认停用
  - 条件模板：气温 ≤ 0℃

- 降水运行
  - 条件：是否降水 = 有降水
  - 驾驶舱初始准备：排风扇超控
  - 驾驶舱准备到滑行：雨刷相关风险提示
  - 滑行：转弯速度低于 7 kt
  - 起飞前：进跑道前雨刷 FAST
  - 起飞后：关雨刷、排风扇解除超控
  - 进近：五边雨刷 FAST
  - 着陆后：雨刷 SLOW
  - 停机：进机位关雨刷

- 跑道代码 444
  - 条件：跑道代码 = 444
  - 起飞：关键风险“跑道代码 444：油门 TOGA”

- 低能见度
  - 条件：能见度 ≤ 400m
  - 驾驶舱准备：易忘项目“LS 按钮开”

- 低能见度 + 侧风限制
  - 条件：能见度 ≤ 400m 且侧风 > 15 kt
  - 起飞前 / 起飞：关键风险“低能见度且侧风大于 15 kt：不能起飞”

- 需要除 / 防冰
  - 条件：需要除 / 防冰 = 需要
  - 起动后：延迟放襟翼直到进跑道
  - 滑行：执行卸冰程序
  - 着陆后：延迟收襟翼直到机位上

### 模拟机科目中心

- 支持新建、编辑、复制、删除科目
- 支持按 21 个阶段维护科目内容
- 支持中文、英文缩写、关键词、别名和拼音首字母搜索
- 支持同时激活多个科目
- 支持标记科目已处置

默认科目模板：

- 双 FMGC 失效
- 地面 LGCIU 1 失效

## 数据保存与同步逻辑

用户数据保存在当前浏览器的 IndexedDB 中。

主要数据：

- 当前飞行进程
- 当前阶段
- 已勾选项目
- 环境条件
- 飞机信息
- 激活的模拟机科目
- 用户维护的规则
- 用户维护的科目库
- 正常流程模板

重要：

- iPad 和电脑的数据不会自动同步。
- iPad 上添加的项目只存在 iPad 浏览器里。
- 电脑网站上添加的项目只存在电脑浏览器里。
- 更新网站代码通常不会删除 iPad 上已经保存的数据。
- 新增默认规则时，应通过 `src/db.ts` 的 bootstrap 逻辑补齐，而不是只在空数据库时写入。
- 如果修改数据结构，要考虑旧数据兼容或迁移。
- 不要轻易更改规则和项目的 `id`，否则可能影响历史勾选状态恢复。

### 手动同步方式

当前同步方式是 JSON 完整备份。

推荐流程：

1. 在内容最新的设备上打开网站。
2. 设置齿轮 → 导出完整备份。
3. 得到 `.json` 文件。
4. 通过 AirDrop、微信文件传输、iCloud、邮箱等方式发到另一台设备。
5. 另一台设备打开网站。
6. 设置齿轮 → 导入备份。
7. 选择合并或覆盖。

建议把 iPad 当作飞行使用主库，并定期导出 JSON 备份。

## 修改规则的建议方式

### 新增一个简单环境规则

通常改 `src/data.ts` 的 `defaultRules`。

一个规则包含：

- `id`：稳定唯一标识，不要随便改
- `name`：用户看到的来源名称
- `enabled`：是否默认启用
- `groups`：触发条件
- `outputs`：命中后注入到哪些阶段
- `updatedAt`

条件逻辑：

- 同一个 group 内全部满足，相当于“并且”
- 多个 group 之间任一满足，相当于“或者”

常用字段：

- `temperature`
- `visibility`
- `windDirection`
- `windSpeed`
- `runwayHeading`
- `crosswind`
- `headwind`
- `precipitation`
- `antiIceRequired`
- `runwayCode`
- `fuelTons`
- `zeroFuelWeightTons`

### 新增环境输入字段

需要同时检查这些位置：

1. `src/types.ts`
   - 给 `EnvironmentData` 增加字段

2. `src/data.ts`
   - 给 `fieldLabels` 增加中文名称

3. `src/App.tsx`
   - 环境字段放在 `EnvironmentModal`
   - 飞机 / 重量字段放在 `AircraftModal`
   - 如果是数字字段，加入 `numericFields`

4. `src/rules.test.ts`
   - 如果字段参与规则判断，最好补测试

### 新增计算字段

例如未来的大重量起飞：

- 可以在 `src/rules.ts` 的 `fieldValue` 中支持派生字段
- 或者先在 `EnvironmentData` 中显式保存

当前已有派生字段：

- `crosswind`
- `headwind`

它们通过风向、风速、跑道方向计算。

## 常见需求示例

### 示例：新增大重量起飞风险

需求可能是：

> 根据飞机信息中的油量和 0 燃油重量计算起飞重量，超过某阈值时，在起飞前或起飞阶段加入风险提示。

推荐做法：

1. 在 `src/rules.ts` 中增加派生字段，例如 `takeoffWeightTons`。
2. 在 `src/types.ts` 的 `EnvironmentField` 允许这个字段。
3. 在 `src/data.ts` 的 `fieldLabels` 增加“起飞重量”。
4. 在 `defaultRules` 中新增规则。
5. 在 `src/rules.test.ts` 中补阈值测试。
6. 在 `src/data.ts` 的 `changelog` 中追加版本记录。

### 示例：新增一个模拟机科目模板

改 `src/data.ts` 的 `defaultSubjects`。

注意：

- 新科目的 `id` 要稳定唯一。
- 如果希望老用户也自动获得新增默认科目，可能需要扩展 `src/db.ts`，像默认规则一样补齐缺失科目。

### 示例：接入云同步

当前没有云同步。如果用户明确要求，可考虑两级方案：

- 轻量方案：GitHub Gist / 云端 JSON，手动上传和下载。
- 完整方案：Supabase / Firebase 登录与数据库同步。

实现前需要明确冲突处理策略，不要默认自动覆盖用户本地数据。
